# Changelog

All notable changes to Brain Bank are documented here.

The format follows [Keep a Changelog](https://keepachangelog.com/en/1.1.0/), and this project aims to follow [Semantic Versioning](https://semver.org/spec/v2.0.0.html) once it leaves pre-release.

Entries are written for operators considering a fork. If you see "Breaking" on a change, upgrading will require action on your side (schema migration, env var addition, config edit). Anything else is a quiet improvement.

## [Unreleased]

### Added

- **`SECURITY.md` at the repo root** documents the vulnerability disclosure path, threat model, and supported-versions policy. Reports route through GitHub Security Advisories at `/security/advisories/new` so the report stays private until a fix is ready. Includes the operator-trust model: Brain Bank is designed for a single trusted operator with `MCP_ACCESS_KEY` as a shared secret; multi-tenant deployments need an upstream enforcement layer (Cloudflare Worker, API Gateway). Why this matters: GitHub auto-surfaces `SECURITY.md` in a dedicated tab on public repos and prompts security researchers toward the private-advisory flow instead of public issues, cutting the most common path for accidental zero-day disclosure.

- **Dashboard merged into the monorepo at `dashboard/` via `git subtree add --squash`.** The Next.js dashboard (formerly hosted in a separate private dashboard repo) now lives inside Brain Bank as a single squashed import rather than a submodule, so a `git clone` of Brain Bank pulls the full dashboard tree in one shot. Prior dashboard history is collapsed into a single squash commit. Why this matters: operators no longer need a second repo permission or a second clone step to stand up the web UI, and future dashboard changes land as normal commits on the monorepo instead of cross-repo coordination. Follow-up work in upcoming Unreleased entries will replace the hardcoded backend URL in `dashboard/src/lib/openBrainApi.ts` with an env var, add the dashboard env vars to `.env.example`, and strip operator-specific branding from the chat system prompt.

### Changed

- **REST `/capture` now honors an optional `source` field in the request body.** Prior behavior hardcoded `metadata.source = "chatgpt"` on every REST capture regardless of caller, because REST originally only served the ChatGPT custom GPT. The endpoint now reads `body.source` when the caller provides it (a non-empty string) and falls back to `"chatgpt"` when absent, so the existing ChatGPT GPT keeps working byte-identically without change. Why this matters: downstream capture sources such as the Gmail bridge, calendar sync, voice memos, and Notion sync can now tag their captures with accurate source values (`gmail`, `gcal`, `voice`, `notion-sync`) instead of masquerading as ChatGPT, which makes digest filtering and source-specific triage meaningful. File: `supabase/functions/open-brain-mcp/index.ts`, `handleRestCapture`.
- **Dashboard `brainBankApi.ts` now reads the backend URL from `BRAIN_BANK_URL`** instead of a hardcoded Supabase project ref. The module also logs a diagnostic and returns null from `brainFetch` if either `BRAIN_BANK_URL` or `BRAIN_BANK_API_KEY` is unset, so a misconfigured deploy fails loud instead of returning empty chat results with no explanation. Why this matters: a fork can deploy the dashboard against its own Supabase project without editing code, and the env var can be rotated at runtime from the deploy platform (Railway, Vercel, etc.) with no rebuild. File: `dashboard/src/lib/brainBankApi.ts`.
- **`.env.example` Phase 5 placeholder block expanded with three dashboard env vars.** The previous "not enumerated yet" stub is replaced by `BRAIN_BANK_URL` (new this release), `BRAIN_BANK_API_KEY` (the dashboard's copy of `MCP_ACCESS_KEY`, previously silently read with no docs), and `DASHBOARD_PASSWORD` (login password the middleware checks). Each entry carries the same "where to find" plus "WHY" framing as the engine section. Operators who only deploy the engine can skip the whole block. Why this matters: the dashboard is now self-documenting from a single `.env.example`, so a fork can stand up both engine and dashboard without reading source to discover required vars.
- **Dashboard chat system prompt now neutral.** The `SYSTEM_PROMPT` in `dashboard/src/app/api/chat/route.ts` previously named the operator in the agent's self-introduction. Replaced with operator-agnostic phrasing referring to "a semantic memory system." A full grep of `dashboard/src` for author-name, domain, and studio-vocabulary tokens returned zero other hits. Why this matters: the prompt no longer leaks operator identity into chat responses for a fork, and the dashboard chat agent introduces itself in operator-neutral terms rather than as anyone's personal assistant. File: `dashboard/src/app/api/chat/route.ts`.
- **Dashboard branding strings centralized in `dashboard/src/config/app.ts`.** New `APP` const object with six fields: `name` (default "Brain Bank"), `tagline` ("Neural Terminal - Semantic Memory System"), `sidebarSubtitle`, `loginSubtitle`, `chatHeader`, `chatSubtitle`. Seven call-sites across five files now import `APP` and reference these fields instead of string literals: `layout.tsx` metadata (title, description, appleWebApp title), `app/api/chat/route.ts` SYSTEM_PROMPT (twice), `app/chat/page.tsx` header, `app/login/page.tsx` h1 and subtitle, `components/NavSidebar.tsx` logo and subtitle, `components/ChatPanel.tsx` header, `lib/useBrainBankChat.ts` error message. All display sites that render in ALL CAPS apply `.toUpperCase()` at render so the config stays human-readable mixed-case. Why this matters: a fork rebranding the dashboard no longer hunts across multiple files for scattered brand literals; one edit in `config/app.ts` updates every visible surface (browser tab, login page, nav sidebar, chat header and error, OG metadata, LLM system prompt).
- **Dashboard defaults rebranded "Open Brain" → "Brain Bank" for public release.** `APP.name` in `dashboard/src/config/app.ts` flipped from "Open Brain" to "Brain Bank"; `dashboard/public/manifest.json` `name` to "Brain Bank"; `dashboard/package.json` `name` to "brain-bank-dashboard" (lockfile regenerated by `npm install`). Lib filenames and exports renamed for symbol consistency: `dashboard/src/lib/useOpenBrainChat.ts` → `useBrainBankChat.ts` (export renamed `useOpenBrainChat` → `useBrainBankChat`), `dashboard/src/lib/openBrainApi.ts` → `brainBankApi.ts` (diagnostic-log prefix updated to match). Three import sites updated in lockstep: `dashboard/src/app/chat/page.tsx`, `dashboard/src/components/ChatPanel.tsx`, `dashboard/src/app/api/chat/route.ts`. `dashboard/src/app/globals.css` file-header comment retitled "Brain Bank Dashboard". `loginSubtitle` shortened from "Neural Terminal v3.0" to "Neural Terminal" (a fork has no v3 history). `npx tsc --noEmit` passes clean. Env var names also flipped in a follow-up commit to `BRAIN_BANK_URL` and `BRAIN_BANK_API_KEY` for consistency (zero impact on any operator already running the dashboard from the upstream private repo until the Phase 5.7 cutover; the rename is a one-time edit in the Railway variable editor at cutover time). Why this matters: an operator cloning Brain Bank sees consistent "Brain Bank" branding in browser tab, sidebar, login, chat header, PWA install prompt, code symbols, and config keys, with no leftover references to the upstream project name in user-visible surfaces, import paths, or env vars.

Source-code removals: the dashboard client list (`dashboard/src/app/clients/page.tsx`) drops the per-client session count badge; the dashboard client detail page (`dashboard/src/app/clients/[id]/page.tsx`) drops the SUMMARY stats card and the SESSION HISTORY table entirely (Profile + Knowledge Graph + Related Thoughts remain); `brain-digest` drops the "Recent sessions" daily list and the "Upcoming sessions" weekly list; `compile-pages` drops the session-history block from client-page wiki compilation; `open-brain-mcp` drops the entire `log_session` MCP tool, the session-history block from `client_context`, the recent-sessions block from `business_context`, and the session-history block from `full_context` (the `full_context` description is updated to no longer claim it searches sessions). Why this matters: cuts ~140 lines of dead code, removes the only remaining tattoo-vocab leak from the schema, and stops shipping an MCP tool nobody calls.

- **Eight byte-identity Phase 3 verification scripts deleted.** `tests/_shared/byte_identity.ts` (the framework, 109 lines) plus seven per-task scripts (`byte_identity_3_5_ingest.ts`, `byte_identity_3_5_mcp.ts`, `byte_identity_3_6.ts`, `byte_identity_3_7_ingest.ts`, `byte_identity_3_7_mcp.ts`, `byte_identity_3_8_digest.ts`, `byte_identity_3_8_mcp.ts`) totaling 656 lines. These were dev-time scrub-verification harnesses that asserted the Phase 3.5-3.8 genericization sweeps removed the author's personal literals from brain-bank source while keeping it byte-identical against live Open Brain on every other axis. Their job is done. They cannot run for any operator who clones brain-bank: the live Open Brain source path is hardcoded to a specific iCloud-Drive location, the fixture is gitignored (`private/profile.dave.json`), and the assertions key off Phase 3 git state. Kept intact: `tests/_shared/profile_test.ts` (generic Deno tests against the public `profile.valid.json` and `profile.missing-required.json` fixtures, useful for any operator running unit tests).

### Changed

- **`docs/deploy-from-scratch.md` Step 2 gains a free-tier 2-active-project cap heads-up.** Supabase's free plan limits each organization to 2 active projects, so operators with two existing projects under the same org hit a `BadRequestException` on the **New project** click without a clear path forward. The new callout block describes the failure mode and a 3-option decision tree (pause an active project for a free, instant, reversible slot; upgrade to Pro to remove the cap; or delete a project no longer needed) plus the paused-vs-deleted distinction so operators do not accidentally delete a project they wanted to keep. Surfaced during the Phase 6.2 v3 fresh-deploy verification when the throwaway project provisioning blocked on the cap and was resolved by pausing the unrelated project for the duration of the test.

### Fixed

- **`brain-digest` no longer reports `status: "delivered"` when the Slack post failed.** The `postToSlack` helper previously called `chat.postMessage`, logged any error via `console.error`, and returned. The response builder reported `delivered` regardless of whether the message actually landed. This made cron silent failures (invalid bot token, channel not found, not in channel, network blip) invisible to operators reading either the curl response, the `cron.job_run_details` body, or the dashboard. The helper now returns `{ ok: true } | { ok: false; error: <slack-error-code> }`, and the response body sets `status: "delivery_failed"` with `slack_error: <code>` when delivery failed (happy-path response shape unchanged, slack_error only present on failure). HTTP status stays 200 in both cases since the synthesis half (which already wrote the digest to the `digests` table) succeeded. Discovered during the Brain Bank Tier 3 fresh-deploy verification when a manual cron fire reported `delivered` but the message never appeared in the test workspace channel. File: `supabase/functions/brain-digest/index.ts`.
- **Active curl examples now use header auth (`x-brain-key: <key>`) instead of URL-parameter auth (`?key=<key>`).** Brain Bank's REST endpoints accept the access key in three ways (`x-brain-key` header, `Authorization: Bearer <key>`, or `?key=<key>` URL parameter) and the function treats them equivalently, but URL-parameter values appear plaintext in Edge Function request logs. The Tier 3 fresh-deploy verification confirmed `get_logs` returns the full `?key=<your-mcp-access-key>` URL on every request, including 401-failed ones. All active curl examples in `docs/deploy-from-scratch.md` Step 10 + 401 troubleshooter, the Notion-sync routine prompt, the `brain-bank-setup` skill SKILL.md (Step 10 + closing message), and the slack-branch reference (Step 9 digest fire) now use `-H "x-brain-key: <key>"`. The MCP Desktop URL in `docs/troubleshooting.md` Section 2 (which must use `?key=` because Claude Desktop doesn't send custom headers) gets a security heads-up explaining the log-leak and suggesting key rotation if log exports are shared. The educational note in `docs/capture-sources/chatgpt-gpt.md` (which describes that the API accepts all three auth modes) is left unchanged. Why this matters: if you set up Brain Bank and shared a log export with a friend or pasted one into a debugging session, you may have shared a working `MCP_ACCESS_KEY` without realizing it.
- **Documented log-retrieval path no longer points at a non-existent CLI command.** README, `docs/deploy-from-scratch.md`, `docs/slack-setup.md`, `docs/troubleshooting.md`, and the `brain-bank-setup` skill references all instructed operators to run `supabase functions logs <name> --project-ref <ref>` for log diagnosis. That subcommand was removed in Supabase CLI v2.75 (current versions reject it with `unknown command` and do not suggest the Dashboard alternative), so operators following the docs hit a dead end exactly when they needed logs the most. All references replaced with the Supabase Dashboard path: Project → Edge Functions → [function] → **Logs**. The `troubleshooting.md` Section 1 also adds a heads-up about the CLI removal so operators recognize the dead-end if they encounter older external guides. Why this matters: webhook-debug failures (Slack URL verification, 500 WORKER_ERROR, profile.json bundling) all surface in the Edge Function logs, and the prior docs sent operators to a CLI command that has not existed for several CLI minor releases.
- **Five doc-clarity fixes from the Tier 3 fresh-deploy verification (F#1-F#5).** All landed as targeted edits in `docs/deploy-from-scratch.md` and `skills/brain-bank-setup/SKILL.md`, one per finding:
  - **F#1 (`supabase link` password drift):** Step 3 prompt-language hedged for CLI v2.75+, which silently defers the DB password prompt to `supabase db push` (Step 7) instead of asking at link time. Operators on v2.75+ saw `Finished supabase link.` with no prompt and worried they had skipped a step.
  - **F#2 (TextEdit silent dotfile-save fail):** Step 5 and the SKILL.md first-secret teaching block now warn macOS operators away from TextEdit, which silently fails to save `.env` (reports "Save successful" but writes nothing). The shape-check grep catches it downstream, but the prose warning saves a wasted save-and-check cycle.
  - **F#3 (`.env` paste-append duplicates):** SKILL.md per-secret flow now explicitly says "find the existing line and replace its value, don't append a new line" (with the implementation-defined-last-occurrence-wins explanation), and the success path adds a `grep -c '^KEY=' .env` dedup safety net that warns when the count exceeds 1.
  - **F#4 (empty-placeholder secrets):** Step 6 push command now filters empty-value lines (`grep -v '^[A-Z_]*=$'`) before piping to `secrets set --env-file /dev/stdin`, so a fresh `.env` doesn't populate Supabase with cosmetic empty entries for the `SLACK_*` and `NOTION_*` vars the operator skipped.
  - **F#5 (Docker-not-running warning noise):** Step 9 now explicitly documents that the `WARNING: Docker is not running` line on every `supabase functions deploy` is benign expected noise. Brain Bank's bundling has always used the API-side path; installing Docker locally to silence the warning gains nothing.
  Why this matters: each was minutes of operator confusion in a fresh-deploy walkthrough; cumulatively, fifteen-plus minutes of "wait, did I skip something?" friction across the first deploy.
- **Slack setup primary flow now treats manual reinstall as mandatory and warns about bot-token regeneration.** Step 8 of `docs/slack-setup.md` (and the parallel skill reference at `skills/brain-bank-setup/references/slack-branch.md`) previously instructed operators to "reinstall when Slack prompts you" and footnoted the manual-reinstall path under "if Slack does not prompt." In practice the prompt fires inconsistently across scope and event combinations, and operators who only reinstalled when prompted often ended up with green Verified, all-green config, and zero inbound message events delivered. Step 8 now makes manual reinstall (OAuth & Permissions → Reinstall to Workspace) the primary instruction every time, regardless of whether Slack prompts. The same step also gains a heads-up that the Bot User OAuth Token can regenerate during reinstall (Slack does not always rotate it, but does sometimes, especially after scope changes), with a token-prefix comparison recipe so the operator can detect a stale `SLACK_BOT_TOKEN` in `.env` before moving to Step 9. The skill's Step 8 row emits the comparison after reinstall using `grep '^SLACK_BOT_TOKEN=xoxb-' .env | cut -c1-25` (prefix only, never the full token to chat). Why this matters: a stale token after reinstall produces `slack_error: invalid_auth` on every digest fire and silent capture failure on every Slack message, with no error visible until F#12's response shape lands. Now operators are prompted to verify the token is current at the moment they're already in the OAuth & Permissions tab.
- **Slack channel-membership step now points operators at the correct UI tab.** Step 5 of `docs/slack-setup.md` (and the skill's slack-branch reference) previously said "the bot appears as a member of each channel," which led operators to look at the channel-details **Members** tab in modern Slack. In modern Slack the Members tab shows humans only; bots appear under the **Integrations** (sometimes labeled **Apps**) section of the same panel. Operators who didn't find the bot under Members assumed the `/invite` failed when it had actually succeeded, sometimes re-invited multiple times, and once gave up entirely. Step 5 now explicitly directs operators to the Integrations section and notes that bots are not listed under Members.
- **Documented `public.call_edge_function` cron wrapper now passes `timeout_milliseconds := 30000` to both `net.http_post` and `net.http_get`.** pg_net's default request timeout is 5 seconds, but `brain-digest` synthesis runs 7 to 25 seconds depending on the LLM round-trip and `compile-pages` can be similar. Under the 5-second default, every cron fire recorded `Timeout of 5000 ms reached` in `net._http_response.error_msg` even though the Edge Function continued running and finished successfully (Edge Functions run independently of the pg_net client). The cron job appeared to fail when it had actually delivered. The wrapper SQL block in `docs/deploy-from-scratch.md` Step 12 now adds a 30-second client timeout, with a "Why" paragraph explaining the Edge-Function-vs-pg_net independence. The `skills/brain-bank-setup/references/cron-branch.md` rationale section gains a parallel bullet so the skill orchestrator surfaces the same explanation. Why this matters: operators inspecting `net._http_response` for cron diagnostics now see truthful results instead of false-positive timeouts, and the existing `cron.job_run_details.status` of `succeeded` no longer appears to contradict an apparent timeout in the response table.
- **`docs/troubleshooting.md` digest delivery section now reflects the F#12 response shape and adds a Slack inbound silent-fail bullet.** Section 4 ("The digest didn't deliver") was rewritten to lead with the new `delivery_failed` plus `slack_error` response shape introduced by the F#12 fix, with a six-code reference list (`invalid_auth`, `not_in_channel`, `channel_not_found`, `is_archived`, `missing_scope`, `fetch_failed`) and the operator-facing fix for each. The "Manually fire the digest" curl example was swapped from `?key=<...>` URL-param auth to `-H "x-brain-key: <key>"` header auth (a missed surface during the prior auth-mode sweep). A new diagnostic note explains that `cron.job_run_details.return_message` is the SQL row count of the wrapper call, not the Edge Function response body, so a `succeeded` cron status does not prove Slack delivery worked. Section 2 ("Slack capture") gained a sixth bullet for the rare all-config-green-but-zero-inbound-POSTs failure mode, with a narrowing recipe (subscribe to a non-message event like `app_home_opened` and check whether THAT one delivers) and a note that other capture sources (REST, MCP, Gmail, voice, Apple Notes) keep working while Slack inbound is broken. Why this matters: when Slack delivery breaks, operators now have one place to land that explains how to read the truthful response, what each error code means, and which side of the integration (inbound vs outbound) is at fault.

## [0.1.0-pre] - 2026-04-21

First pre-release snapshot. Everything below represents the initial open-sourcing cut of an engine that has been running in production for the author since March 2026. No prior public release, so nothing here is flagged Breaking.

### Added

- **Database schema as 11 migration files** under `supabase/migrations/`. Includes the `thoughts` table (pgvector embedding column and HNSW index), the `digests` archive, the `compiled_pages` wiki table, auto-resolve action item tables, required extensions (pgvector, pg_cron, pg_net, supabase_vault), RLS policies, and the `call_edge_function` vault wrapper that lets pg_cron call Edge Functions without secrets on disk. Running `supabase db push` from a fresh Supabase project replays the full schema.
- **Four Edge Functions** under `supabase/functions/`: `ingest-thought` (Slack plus thread routing plus the auto-resolve pipeline), `open-brain-mcp` (MCP server with 19 tools plus a REST `/capture` endpoint), `brain-digest` (daily and weekly synthesis), and `compile-pages` (wiki compilation engine). All four run on the Deno runtime with no build step.
- **Profile config layer.** A `profile.example.json` schema with a `loadProfile()` helper and Zod validation. Operators copy the example to `profile.json` (gitignored) and edit it once; the engine reads operator identity, capture prefixes, calendar event types, content types, and LLM persona strings from this file at runtime. Three fixtures under `tests/_shared/fixtures/` exercise the valid, missing-required, and tattoo-studio variants. Why this matters: the engine stays neutral in the public repo. A tattoo studio and a software consultancy can share the same codebase without vocabulary from one bleeding into the other's digests.
- **Root-level `deno.json`** with a Deno 2.x `workspace` field pointing at the four function directories. Lets you run `deno check supabase/functions/*/index.ts` from the repo root and get honest type coverage across all four at once.
- **README** rewritten as a fork-and-deploy pitch. Five-step quickstart, "where to find" pointers on prerequisites, concrete error-recovery hints (for example, if `supabase functions deploy` returns 500, run `supabase functions logs <name>`).
- **`.env.example`** at the repo root enumerating all 11 Edge Function env vars, grouped by domain (Supabase, OpenRouter, MCP access key, Slack, Notion). Each variable carries a "where to find" pointer. Non-obvious ones also carry a "WHY" line. Dashboard env vars are flagged as Phase 5 placeholders and will land when the dashboard merges into the monorepo.

### Changed

- **All four Edge Functions genericized from the author's personal fork.** Six sweep passes across `brain-digest`, `compile-pages`, `open-brain-mcp`, and `ingest-thought` replaced hardcoded tattoo-studio vocabulary, operator emails, mechanical-capture prefixes, and enum values (event types, content types, client event types) with reads from `profile.json`. A final prose-neutralization pass caught framing strings inside LLM prompt templates. Behavior is byte-identical against the author's `profile.json`, and neutral against the shipped example.
- **TypeScript baseline cleaned.** 13 real errors across `brain-digest` and `ingest-thought` cleared via three small type-only edits (row-type broadening, lambda parameter annotations, an `EdgeRuntime` ambient declaration). `deno check` from the repo root now reports zero errors across all four functions.
- **Live production Edge Functions now deploy from the Brain Bank repo** instead of the author's private working archive. Versions after the swap: `ingest-thought` v30, `open-brain-mcp` v33, `brain-digest` v26, `compile-pages` v10. The cutover happened on 2026-04-21 and is being validated through a parallel drift window (silent-observer `-v2` slugs fed from the private archive, compared against the live brain-bank outputs).

### Fixed

- **Edge Function deploys now include `profile.json` in the upload bundle.** The first attempted brain-bank swap returned 500 WORKER_ERROR because the Supabase CLI bundler only uploads files reachable through the import graph, and the earlier runtime `Deno.readTextFileSync` approach left `profile.json` outside that graph. Replaced with a JSON module import (`import profileDefaults from "./profile.json" with { type: "json" }`), which the bundler recognizes as a module-graph edge. A throwaway test function verified the fix before the retry deploy. Why this matters: if you build anything that reads a local file from an Edge Function, make it a module import, not a filesystem read.

### Notes for future readers

- This is a pre-release. The engine is live for the author, but the deploy-from-scratch walkthrough (`docs/deploy-from-scratch.md`) is still being written. The first tagged release (`v0.1.0`) ships once a friend deploys successfully from a cold clone using only the shipped docs.
- The dashboard (Next.js on Railway) is still a separate private repo. It will merge into `dashboard/` via `git subtree add` during Phase 5, after which dashboard env vars get added to `.env.example`.
